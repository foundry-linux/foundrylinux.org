# Plan: Package xa65 as a real `.deb` (first non-meta package)

**Date:** 2026-05-18
**Status:** Done (amd64; arm64 deferred)

## Context

`foundry-apt` currently ships five `Architecture: all` metapackages and zero real upstream packages. The Phase 0 retro-tools installer ([`foundry-linux-setup/install-foundry-linux-retro-tools.sh:137-142`](../../foundry-linux-setup/install-foundry-linux-retro-tools.sh)) source-builds five tools into `~/opt/`: `xa65`, `f9dasm`, `libvgm`, `vgmstream`, and downloads Ghidra. None are real `.deb`s yet, so the `foundry-linux-retro-tools` metapackage can't `Depends:` on them — meaning the Phase 0 → Phase 1 "collapse" rehearsed in [`docs/plans/2026-05-17-per-metapackage-install-scripts.md`](2026-05-17-per-metapackage-install-scripts.md) is blocked on this.

**xa65 first** because it's the simplest of the five (small C codebase, mature Makefile with `install` target, GPL-2.0, no native deps beyond libc) and `packages/xa65/` already exists as an empty placeholder directory.

## Vendoring decision

| Field | Value |
|---|---|
| Upstream | <https://github.com/fachat/xa65> (André Fachat's repo; mirrors releases from canonical Cameron Kaiser at <http://www.floodgap.com/retrotech/xa/>) |
| Tag | `xa-2.4.1` |
| Tarball URL | `https://github.com/fachat/xa65/archive/refs/tags/xa-2.4.1.tar.gz` |
| License | GPL-2.0-only (per `xa/COPYING`) |
| Build root | `xa/` subdir (`make`, `make install`) |
| Binaries produced | `xa`, `reloc65`, `ldo65`, `file65`, `printcbm`, `uncpk` (6 binaries) |
| Man pages | `xa.1`, `reloc65.1`, `ldo65.1`, `file65.1`, `printcbm.1`, `uncpk.1` |
| Architectures | `amd64`, `arm64` (pure C, built per arch on CI runner) |
| Debian version | `2.4.1-1foundry1` (upstream `2.4.1`, packaging revision `1foundry1`) |

**Why fachat over floodgap:** GitHub archive URLs are content-addressable by tag → easy sha256 pin (matches `packages/task/build.sh` pattern that was previously in this repo). Floodgap is HTML-scraped — fragile. Same code per fachat's README ("mirroring Cameron's releases").

## `build.sh` shape

```bash
#!/usr/bin/env bash
# Build xa65 (6502/65816 cross-assembler, Fachat/Kaiser) as Debian .debs for
# amd64 and arm64 from the upstream xa-2.4.1 release.

set -euo pipefail

UPSTREAM_VERSION="${XA65_VERSION:-2.4.1}"
TAG="xa-${UPSTREAM_VERSION}"
DEB_REVISION="1foundry1"
DEB_VERSION="${UPSTREAM_VERSION}-${DEB_REVISION}"

# sha256 of github.com/fachat/xa65/archive/refs/tags/xa-${TAG}.tar.gz (verified once)
SHA256_TARBALL="${XA65_SHA256:-<PIN AT BUILD TIME>}"

# (per-arch build_one similar to packages/task/build.sh, but:
#   - source build, not binary repack
#   - `make -C "$src/xa"` then `make -C "$src/xa" install DESTDIR=$stage prefix=/usr`
#   - control file with Depends: libc6, Architecture: ${arch}, Section: devel)
```

Native arch only: CI runners are amd64. arm64 build uses `make` with `CC="aarch64-linux-gnu-gcc"` cross toolchain (Debian package `gcc-aarch64-linux-gnu`). Confirm install-deps in `publish.yml`.

    Expect: control fields show `Package: xa65`, `Version: 2.4.1-1foundry1`, `Architecture: amd64`, `Section: devel`, `Depends: libc6 (>= ...)`.     Expect: `xa` reports version `2.4.1`; all six binaries are on PATH.

7. **Metapackage wiring.** After bumping `foundry-linux-retro-tools` to `1.0.1` with `xa65` in `Depends:`:

    ```

    dpkg-deb -I dist/foundry-linux-retro-tools_1.0.1_all.deb | grep -i depends
    ```

    Expect: `Depends:` line contains `xa65`.

8. **Phase 0 collapse (deferred).** Stripping the xa65 source-build is blocked on Phase 0 first adding `apt.foundrylinux.org` to `sources.list`. Tracked as a separate TODO ("Phase 0 configures foundry-apt as a source"). For this plan, the source-build remains as the bootstrap path.

## Out of scope (next plans)

- Packaging `f9dasm`, `libvgm`, `vgmstream` — same pattern but per-tool quirks.
- Packaging Ghidra — much heavier (Java, ~400 MB); probably its own metapackage `foundry-linux-ghidra` because it dominates the size.
- Cross-compile arm64 — start with native amd64-only; add arm64 once amd64 ships, in a follow-up.

## Open questions (decide while implementing)

- **Cross-compile or amd64-only first?** The Phase 0 script doesn't care about arm64; the apt repo `Architecture: all` metapackages don't either. Cleanest first step: ship amd64-only, add arm64 in a follow-up. Reduces the surface for this plan.
- **`xa65` package name vs `xa`.** Upstream binary is `xa`, but `xa` is too generic for an apt package name. Debian convention is to name after the project (`xa65`), matching what `~/opt/xa65/` already implies.
